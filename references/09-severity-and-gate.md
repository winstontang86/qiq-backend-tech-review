# 严重程度、评分与上线门禁

> **本文件是所有校验者的共同基准**。任何校验者输出问题时，最终严重程度必须落到本文档定义的 5 档之一；最终报告的"是否可进入下一阶段"结论必须由本文档的门禁规则推导得出，而非主观判断。

## 一、统一严重程度（5 档）

所有校验者对外输出问题时，必须统一使用以下 5 档严重程度：

| 等级 | 含义 | 处置要求 |
|---|---|---|
| **Blocker** | 阻塞性问题：上线/进入下一阶段必须修复，不可豁免（除非降级为 High 并写明补救） | 必须修复 |
| **High** | 高风险问题：上线前必须修复或正式豁免（带 owner、补救方案、复盘时间） | 必须修复 / 正式豁免 |
| **Medium** | 中风险问题：建议修复，不修复需评估业务影响 | 建议修复 |
| **Low** | 低风险问题：长期优化项 | 可选优化 |
| **Info** | 提示性建议、设计亮点反馈、可讨论项 | 仅提示 |

## 二、各校验者原始等级 → 统一等级映射

各校验者 reference 中可保留原有领域等级（更利于该领域表达），但**最终汇总到报告时必须按下表映射**：

| 校验者 / 原始等级 | 映射 |
|---|---|
| C-Critical（一致性） | Blocker |
| C-Major | High |
| C-Minor | Medium |
| C-Suggestion | Low / Info |
| HA-P0（容错） | Blocker |
| HA-P1 | High |
| HA-P2 | Medium |
| HA-P3 | Low |
| PERF-High（性能） | High |
| PERF-Medium | Medium |
| PERF-Low | Low |
| SEC-Critical（安全） | Blocker |
| SEC-High | High |
| SEC-Medium | Medium |
| SEC-Low | Low |
| DATA-P0（数据） | Blocker |
| DATA-P1 | High |
| DATA-P2 | Medium |
| DATA-P3 | Low |
| OPS-High（运维） | High |
| OPS-Medium | Medium |
| OPS-Low | Low |
| SIMP-High（简洁性） | Medium（简洁性问题原则上不阻塞上线） |
| SIMP-Medium | Low |
| SIMP-Low | Info |

> 注意：简洁性维度的问题**永远不映射为 Blocker/High**——过度设计是技术债，但不会阻塞上线。

## 三、Blocker 自动判定规则

满足任一条件，问题必须升级为 **Blocker**（即使原始等级低于 Critical/P0）：

1. **正确性类**
   - 核心业务逻辑存在矛盾，且方案中未给出消歧方案。
   - 核心需求（FR-xxx）在设计中完全无对应实现。
   - 核心状态机存在死状态、不可达状态或非法转换路径。

2. **资金/账务类**
   - 涉及金额计算的字段使用了浮点类型。
   - 资金相关操作的并发场景下可能出现超卖、超扣、双花、错账。
   - 资金相关数据无对账、无补偿、无审计日志。

3. **数据丢失类**
   - 核心业务数据无备份方案、无 RPO 定义。
   - 数据迁移无回滚方案。
   - 异步链路（消息/任务）的关键路径无重试、无死信、无可恢复机制。

4. **安全类**
   - 任何已知 SEC-Critical 漏洞（如未鉴权的写接口、明文存储凭证、SQL 注入）。
   - 个人敏感信息明文存储或在日志中明文输出。
   - 越权风险存在且无防御。

5. **合规类**
   - 违反明确法规要求（如个人信息保护法对授权/删除/导出的强制要求）。
   - 未经授权的数据跨境传输。

6. **可用性 vs SLA 错位**
   - 声称 99.99% 可用性但单机房单实例部署。
   - 声称强一致但仅做了异步复制。
   - 声称 RPO=0 但备份是 T+1。

7. **外部假设错位 / 与既有约定冲突**（防"基础假设崩塌"类事故）
   - 方案对外部依赖（中间件 / 平台 / 既有服务）的能力假设，与该依赖在**本系统实际部署形态**下提供的语义不一致，且方案的关键路径建立在该假设之上。
   - 方案使用了仓库代码 / 注释 / 架构规范中已**显式标记为废弃 / 不推荐 / 不可靠**的能力，且未给出"为何仍要使用 + 如何规避原始问题"的说明。
   - 方案**首次引入**仓库 / 团队从未使用过的中间件 / 命令族 / 机制作为关键路径依赖，但 Step 1.5 未执行或未输出尽调结论。
   - **判定要点**（任一不满足即升 Blocker）：
     1. 关键路径下"丢一次事件 / 偏一次语义"是否可自愈？
     2. 是否提供了不依赖该外部能力的兜底（reconcile / 全量重同步 / 启动期权威源拉取 / 对账修复）？
     3. 方案是否**显式列出了**所有关键语义假设并逐条与本系统实际形态对照过？

## 四、上线门禁规则（最终结论由此推导）

最终报告"是否可进入下一阶段"的结论**不允许凭主观判断**，必须按以下规则机械推导：

### 4.1 进入编码阶段（Design → Coding）

允许进入，需满足：

- [ ] 无 Blocker。
- [ ] 一致性与完整性维度：核心需求-设计映射完整、核心流程闭环、核心接口契约定义完整。
- [ ] 关键数据模型已定义，存储选型已明确。
- [ ] 允许存在性能、运维细节待补充。

### 4.2 进入测试阶段（Coding → Testing）

允许进入，需满足：

- [ ] 无 Blocker。
- [ ] 所有 High 级数据一致性、安全问题已有明确修复方案（不要求已实现）。
- [ ] 灰度策略、回滚方案、监控告警方案已明确。
- [ ] 容量评估已完成。

### 4.3 进入上线阶段（Testing → Production）

允许进入，需满足全部以下条件：

- [ ] 无 Blocker。
- [ ] 核心系统的所有 High 级问题已修复或正式豁免（豁免必须带 owner + 补救方案 + 复盘时间）。
- [ ] 安全、资金一致性、数据丢失类问题**不允许豁免**，必须修复。
- [ ] 有可执行的回滚方案，且回滚时间满足 RTO。
- [ ] 监控告警覆盖核心 RED 指标 + 核心业务指标。
- [ ] 资金/核心数据有对账或补偿机制。
- [ ] 应急预案与 on-call 安排到位。

### 4.4 结论模板

```
**结论**：{允许 / 不允许} 进入 {编码 / 测试 / 上线} 阶段。

**判定依据**：
- Blocker 数量：N（列出编号）
- 未修复且未豁免的 High：N（列出编号）
- 未满足的门禁项：（列出未勾选的检查项）

**阻塞项**：（如不允许进入下一阶段，逐一列出阻塞原因）
```

## 五、豁免机制（Waiver）

**仅 High 及以下可豁免，Blocker 不可豁免**。

### 5.1 不可豁免的红线（即使原本是 High 也不允许豁免）

- 安全漏洞（任何 SEC-High 及以上）
- 资金/账务一致性问题
- 用户敏感数据保护问题
- 合规强制要求
- 核心数据丢失风险

如需"暂不修复"，必须先把问题降级讨论是否真的属于 High，而不是直接豁免。

### 5.2 豁免必须填写

```
**豁免编号**：W-xxx
**对应问题**：[原问题编号]
**豁免理由**：为何当前阶段不修复
**风险接受人**：（必须是有授权的负责人，如技术负责人/业务负责人）
**补救方案**：在不修复的情况下如何降低风险（监控/降级/限流/人工兜底）
**复盘时间**：何时重新评估（必须给出明确日期，不允许"待定"）
**触发回滚条件**：满足什么条件后必须立即修复
```

## 六、评分 Rubric（1–10）

各维度评分必须按以下 Rubric 给出，避免主观漂移：

| 分数 | 含义 | 判定标准 |
|---|---|---|
| 9–10 | 优秀 | 设计完整闭环，仅有少量非关键优化建议（仅 Low/Info） |
| 7–8 | 良好 | 主路径完整，存在少量 Medium，无 High 及以上 |
| 5–6 | 合格 | 可实现，但关键非功能性设计存在缺口（≤2 个 High，无 Blocker） |
| 3–4 | 不合格 | 存在多个 High 或核心设计缺失，**不建议进入下一阶段** |
| 1–2 | 严重不合格 | 存在 Blocker 或核心逻辑/安全/数据一致性严重风险 |

### 6.1 强制约束

- **存在 Blocker → 该维度评分不得 ≥ 4**。
- **存在未豁免 High → 该维度评分不得 ≥ 7**。
- **整体评分不是各维度算术平均值**：
  - 任一维度有 Blocker → 整体结论为"不允许进入下一阶段"，整体评分不得 ≥ 4。
  - 安全、资金一致性维度评分 < 6 → 整体评分不得 ≥ 6。
- 简洁性维度评分低（即"过于复杂"）**不影响**进入下一阶段，但应进入"建议优化"清单。

## 七、各校验者统一 Output Format（所有 reference 共用）

各校验者 reference 的 Output Format 一律采用本节模板，**仅替换编号前缀**与少量维度专属字段；不再各自重复定义。

### 7.1 通用问题模板

```
### [<前缀>-编号] [问题类型] [严重程度: <原始等级> → 归一化: Blocker/High/Medium/Low/Info]

**位置**：方案第 X 章 / 接口 / 表 / 链路 / 组件
**原文证据**：引用方案原文片段；如未提及，写"方案未说明"
**判断依据**：与哪条上下文 / SLA / 一致性原则 / 暴露面 / 业务重要性相关
**问题描述**：具体描述发现了什么问题
**修改建议**：具体的修改方案（必须可执行；必修项需含验收标准）
```

### 7.2 编号前缀与维度专属可选字段

| 校验者 | 前缀 | 原始等级集 | 维度专属可选字段（按需追加在"修改建议"前） |
|---|---|---|---|
| 1 一致性 | C- | Critical/Major/Minor/Suggestion | — |
| 2 容错 | HA- | P0/P1/P2/P3 | 影响分析（功能/范围/持续时间）、优先级说明 |
| 3 性能 | PERF- | High/Medium/Low | 量化分析（当前预估/瓶颈阈值/到达时间）、优化效果预估、实现成本 |
| 4 安全 | SEC- | Critical/High/Medium/Low | 攻击场景、影响、参考标准（OWASP/国标/行业） |
| 5 数据 | DATA- | P0/P1/P2/P3 | 场景还原（可复现操作序列）、影响、trade-off |
| 6 运维 | OPS- | High/Medium/Low | 影响场景、优先级说明 |
| 7 简洁性 | SIMP- | High/Medium/Low（信心度: 确定/较确定/需讨论） | 当前设计、复杂度成本、简化方案、简化后 trade-off、前提条件、失效条件（必填） |
| 8 API 兼容 | API- | Blocker/High/Medium/Low/Info | 风险场景、回滚影响（可回滚/不可回滚/需反向迁移） |

### 7.3 最终汇总报告中的呈现格式

```
[原编号 如 SEC-3 / HA-2] [统一严重程度 Blocker/High/Medium/Low/Info]
位置 / 原文证据 / 判断依据 / 修改建议 / Owner（可选）/ 验收标准（必修项必填）
```
