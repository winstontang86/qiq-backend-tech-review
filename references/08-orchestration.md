# 编排与仲裁

## 一、校验者分组策略

根据方案类型，不是每次都需要全部 7 个校验者。

### 必选组（每个方案都需要）
- 校验者 1：一致性与完整性
- 校验者 7：简洁性（奥卡姆剃刀）

### 按方案特征选择

| 方案特征 | 额外启用的校验者 |
|---|---|
| 分布式系统 / 微服务 | 校验者 2（容错）+ 校验者 5（数据一致性） |
| 高并发 / 大数据量 | 校验者 3（性能）+ 校验者 5（数据一致性） |
| 面向公网 / 涉及用户数据 | 校验者 4（安全） |
| 已有系统改造 / 长期维护 | 校验者 6（可运维性） |
| 核心交易 / 资金系统 | 全部启用 |

### 按业务重要性调整深度

| 业务重要性 | 校验深度 |
|---|---|
| 核心系统 | 全部校验者，严格模式 |
| 重要系统 | 必选组 + 2~3 个按需校验者 |
| 一般系统 | 必选组 |
| 内部工具 / MVP | 仅校验者 1（轻量模式）+ 校验者 7 |

## 二、冲突仲裁规则

当不同校验者建议冲突时，按以下优先级决策：

### 优先级排序

1. **正确性**（校验者 1）> 一切
   - 逻辑矛盾和遗漏必须修复，无论是否增加复杂度

2. **安全性**（校验者 4）> 简洁性
   - 安全漏洞必须修复，即使增加复杂度

3. **数据正确性**（校验者 5）> 性能 > 简洁性
   - 涉及 "钱" 的数据一致性，不能为了性能或简洁而妥协

4. **需求驱动的健壮性**（校验者 2）> 简洁性（校验者 7）
   - 如果 SLA 明确要求 99.99%，必要的容错机制不能因为 "简化" 而去掉

5. **简洁性**（校验者 7）> 非需求驱动的健壮性 / 性能
   - 如果 SLA 只要求 99.9%，不需要为极端场景增加大量复杂度
   - 如果当前 QPS 100，不需要为 QPS 100000 做优化

### 量化判断标准

- 故障概率 × 故障影响 > 应对方案的长期维护成本 → 采纳健壮性建议
- 故障概率 × 故障影响 < 应对方案的长期维护成本 → 采纳简洁性建议

## 三、上下文注入模板

每次调用校验者时，把以下上下文显式注入：

### 业务信息
- 业务重要性：{核心 / 重要 / 一般}
- 业务领域：{电商 / 金融 / 社交 / 内部工具 / ...}
- 用户规模：{DAU / MAU}
- 是否涉及资金：{是 / 否}

### 技术信息
- 团队规模：{X 人}
- 团队技术栈熟悉度：{列出团队熟悉的技术栈}
- 现有基础设施：{列出已有的中间件 / 平台}
- 期望上线时间：{X 周后}

### 系统规模
- 当前 QPS：{xxx}
- 预期 1 年后 QPS：{xxx}
- 当前数据量：{xxx}
- 预期 1 年后数据量：{xxx}

### SLA 要求
- 可用性目标：{99.9% / 99.95% / 99.99%}
- 核心接口延迟要求：{P99 < xxx ms}
- 数据一致性要求：{强一致 / 最终一致，最终一致的窗口}
- RPO：{xxx}
- RTO：{xxx}
