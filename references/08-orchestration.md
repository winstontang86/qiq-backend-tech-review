# 编排与仲裁

> 本文件定义校验者选择策略、冲突仲裁规则、上下文注入模板。**严重程度归一化、上线门禁、评分 Rubric** 见 [@references/09-severity-and-gate.md](09-severity-and-gate.md)。

## 一、校验者分组策略

根据方案类型和特征，不是每次都需要全部 8 个校验者。

### 必选组（每个方案都需要）
- 校验者 1：一致性与完整性
- 校验者 7：简洁性（奥卡姆剃刀）

### 按方案特征选择

| 方案特征 | 额外启用的校验者 |
|---|---|
| 分布式系统 / 微服务 | 校验者 2（容错）+ 校验者 5（数据一致性） |
| 高并发 / 大数据量 | 校验者 3（性能）+ 校验者 5（数据一致性） |
| 面向公网 / 涉及用户数据 | 校验者 4（安全） |
| 已有系统改造 / 长期维护 | 校验者 6（可运维性）+ 校验者 8（API 契约与兼容性） |
| 接口设计 / Schema 演进 / 滚动发布 | 校验者 8（API 契约与兼容性） |
| 核心交易 / 资金系统 | 全部启用 |

### 按业务重要性调整深度

| 业务重要性 | 校验深度 |
|---|---|
| 核心系统 | 全部 8 个校验者，严格模式 |
| 重要系统 | 必选组 + 3~4 个按需校验者 |
| 一般系统 | 必选组 + 1~2 个按需校验者 |
| 内部工具 / MVP | 仅校验者 1（轻量模式）+ 校验者 7 |

## 二、冲突仲裁规则

当不同校验者建议冲突时，按以下优先级决策：

### 优先级排序

1. **正确性**（校验者 1）> 一切
   - 逻辑矛盾和遗漏必须修复，无论是否增加复杂度

2. **安全性**（校验者 4）> 简洁性
   - 安全漏洞必须修复，即使增加复杂度

3. **数据正确性**（校验者 5）> 性能 > 简洁性
   - 涉及"钱"的数据一致性，不能为了性能或简洁而妥协

4. **需求驱动的健壮性**（校验者 2）> 简洁性（校验者 7）
   - 如果 SLA 明确要求 99.99%，必要的容错机制不能因为"简化"而去掉

5. **兼容性**（校验者 8）> 性能 / 简洁性
   - 破坏性变更必须先解决；不能为了性能或简洁牺牲新旧版本共存的稳定性

6. **简洁性**（校验者 7）> 非需求驱动的健壮性 / 性能
   - 如果 SLA 只要求 99.9%，不需要为极端场景增加大量复杂度
   - 如果当前 QPS 100，不需要为 QPS 100000 做优化

### 量化判断标准

- 故障概率 × 故障影响 > 应对方案的长期维护成本 → 采纳健壮性建议
- 故障概率 × 故障影响 < 应对方案的长期维护成本 → 采纳简洁性建议

### 不可妥协的红线

以下问题不参与"成本 vs 收益"的折衷，必须修复：

- 任何 SEC-Critical 安全漏洞（详见 [@references/09-severity-and-gate.md](09-severity-and-gate.md) Blocker 规则）
- 涉及金额计算的浮点类型使用
- 资金 / 账务相关的并发场景一致性问题
- 核心数据丢失风险（无备份 / 无 RPO 定义）
- 合规强制要求

## 三、严重程度归一化

各校验者 reference 内可保留各自领域习惯的等级表达（C-Critical / HA-P0 / SEC-High 等），最终汇总到报告时**必须按 [@references/09-severity-and-gate.md](09-severity-and-gate.md) 第二节的映射表统一为**：

`Blocker / High / Medium / Low / Info`

报告中每个问题应同时呈现"原始编号 + 原始等级 + 归一化等级"，便于追溯与决策。

## 四、上下文注入模板

每次调用校验者时，把 [@references/00-context-template.md](00-context-template.md) 中"业务信息 / 技术信息 / 系统规模 / SLA 要求 / 执行模式"五块内容显式注入当前校验者的视角，不重复列字段。
