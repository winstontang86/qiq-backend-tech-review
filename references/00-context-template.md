# 审查上下文模板

> 每次启动 review 前先填好此表，缺失项标 "未说明" 并向用户追问（**正式评审模式**）；初步扫描模式下可临时假设并继续。

- 模式：{正式评审 / 初步扫描}
- 上下文置信度：{高 / 中 / 低}
- 关键假设（初步扫描模式必填，逐条列出影响结论的假设）：

## 业务信息
- 业务重要性：{核心 / 重要 / 一般}
- 业务领域：{电商 / 金融 / 社交 / 内部工具 / ...}
- 用户规模：{DAU / MAU}
- 是否涉及资金：{是 / 否}

## 技术信息
- 团队规模：{X 人}
- 团队技术栈熟悉度：{列出团队熟悉的技术栈}
- 现有基础设施：{已有的中间件 / 平台}
- 期望上线时间：{X 周后}

- 当前 QPS：{xxx}

- 当前数据量：{xxx}

## SLA 要求
- 可用性目标：{99.9% / 99.95% / 99.99%}
- 核心接口延迟要求：{P99 < xxx ms}
- 数据一致性要求：{强一致 / 最终一致，最终一致的窗口}
- RPO：{xxx}（可容忍数据丢失时间：0 / 秒级 / 分钟级）

## 关键缺项追问

**正式评审模式**下，如果用户没提供以下任一项，必须追问后再进入审查：

1. 业务重要性等级
2. 是否涉及资金
3. 当前 QPS 量级
4. SLA 可用性目标

**初步扫描模式**下，可在以上四项缺失时临时假设并继续，但必须在最终报告头部以"假设与置信度"小节明确登记：

| 项 | 状态 | 说明 |
|---|---|---|
| 明确事实 |  | 方案中明确写明的内容 |
| 审查时假设 |  | 因方案未说明而临时采用的假设（逐条列出） |
| 未知项 |  | 影响结论但既未明确也未假设的内容 |
| 结论置信度 | 高/中/低 | 整体置信度评估 |
