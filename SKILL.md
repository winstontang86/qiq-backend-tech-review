# qiq-backend-tech-review — 互联网后台技术方案多维校验

## When to use

启用条件（任一即可）：

- 用户给出后台 / 分布式 / 微服务 / 数据系统 / 高并发系统的技术方案 / 设计文档 / RFC，希望做"评审 / review / 挑刺 / 找漏洞 / 上线前检查"。
- 用户希望对一份方案做**多维度系统化检查**，而非单点提问（如只问性能）。
- 用户使用 `qiq-backend-tech` skill 产出了技术方案，希望进入"评审 → 修订"闭环。本 skill 与之配对：spec skill 输出的 FR / NFR / ASMP 编号、章节结构、关键存储与数据结构清单是本 skill 的对账锚点。

不适用：

- 单点技术问题（如"Redis 为什么慢"）——直接回答即可。
- 前端 / 移动端 / 算法模型方案——本 skill 维度不匹配。
- 业务需求 / PRD 评审——本 skill 关注技术实现方案。

## 执行模式（先与用户对齐再开始）

二选一，用户未明确说明时默认"正式评审"，并把当前模式告诉用户、允许切换：

| 模式 | 触发 | 上下文要求 | 是否输出门禁结论 | 适合场景 |
|---|---|---|---|---|
| **正式评审**（默认） | 默认；用户希望严谨 review | 缺关键 4 项必须追问后再开始（业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标） | 是 | 上线前评审、正式 design review、RFC 评审 |
| **初步扫描** | 用户明确表达"先粗看 / 快速过一遍 / 不用追问" | 允许临时假设继续，但报告头部必须声明置信度、关键假设、需复核结论 | 否，仅输出风险清单 | 初稿自检、早期讨论、需求方快速对齐 |

## Workflow

### Step 1 — 收集审查上下文（必须）

**禁止跳过此步直接 review**：核心系统和内部工具不能用同一把尺子。

#### 1.0 前置文件预读（强制，先于任何评审动作）

> 目的：评审的依据**不只是技术方案文本**，还包括方案上游的"原始诉求"。如果方案漏抽 / 误读 / 静默删减了原始诉求里的功能点，单看方案永远发现不了——必须用原始记录回溯对账。

**预读文件清单（在 Step 1.x 任何检查动作开始前完成）**：

> `TECH_DESIGN_DIR` 解析规则统一见 §7.1.0：**`TECH_DESIGN_DIR = <原仓库根目录>/.tech-design/`**（仓库根 = 从原方案文件所在目录逐级向上回溯遇到的第一个含 `.git`/`.hg`/`.svn` 标识的目录；粘贴文本 / 无仓库根场景以当前工作目录兜底）。本步先按该规则解析出 `TECH_DESIGN_DIR`，再读取其下文件。

| 优先级 | 文件路径 | 来源 | 用途 | 缺失时处理 |
|---|---|---|---|---|
| P0 | `<TECH_DESIGN_DIR>/requirements.md` | `qiq-backend-tech` 技术方案生成 skill 在产出方案前与用户对话采集的**需求采集原始记录**（一问一答 / 业务背景 / 显式与隐式诉求 / 边界澄清 / 拒绝项） | **FR 源头之一**（与方案内"需求章节"并列、且优先级更高）；用于发现"方案对原始诉求的漏抽 / 改写 / 静默删减" | 文件不存在直接跳过本条；**不**追问、**不**报错；在最终报告头部"原始需求记录路径"字段填"无（未发现 .tech-design/requirements.md）" |
| P0 | 原方案文件本身 | 用户提供 | 评审主体 | 必须存在；不存在则评审无法进行 |
| P1 | `<TECH_DESIGN_DIR>/intermediates/*.md`（如本 skill 历史跑产生的中间产物） | 本 skill 上一次跑的中间产物 | 重跑场景下避免重复检索仓库规则 / 重新核查外部假设；可作为对照基线 | 不存在直接跳过 |

**硬动作**：

1. 按 §7.1.0 解析 `TECH_DESIGN_DIR`（固定在仓库根；无仓库根时按兜底规则）；本次解析结果作为本评审全局变量，Step 7 落盘时直接复用，**不得**重新解析到不同路径。
2. 检测 `${TECH_DESIGN_DIR}/requirements.md` 是否存在；若存在，**真正调用读文件工具读取其完整内容**，作为后续 Step 1 / Step 4 校验者 1 的输入。
3. 在最终报告头部"原始需求记录路径"字段填写：存在则写绝对路径；不存在则写"无（未发现 .tech-design/requirements.md）"。
4. 若 `requirements.md` 与方案文本出现**功能点 / 业务约束 / 显式拒绝项**层面的不一致，按下文"不一致处置"操作；**不得**默认相信方案、忽略原始记录。

**优先级原则（核心）**：

- **原始诉求 ≥ 方案文本**：当 `requirements.md` 中明确写明的功能点 / 拒绝项与方案不一致时，默认相信原始记录；方案需要给出"为什么改 / 是否经过用户确认"的显式说明，否则视为方案漏抽 / 静默删减。
- `requirements.md` 中标记为"显式拒绝 / 不做 / 暂不支持"的项，方案中**不应**出现对应实现；如出现，记一条 C-Major（"超目的实现"）。
- `requirements.md` 中标记为"必须 / 强诉求 / P0"的项，方案中**必须**有对应落点；缺失则记一条 C-Critical 候选（最终是否升级 Blocker 由 Step 6 门禁判定）。

**不一致处置**：

- 缺漏（requirements.md 有，方案无）→ 校验者 1（§2.1 功能完整性矩阵）按 ❌ 处理，并在"原文证据"列引用 `requirements.md` 中的对应段落。
- 改写（requirements.md 与方案描述同一功能但语义偏移）→ 校验者 1 按 🟡 处理，并在"判断依据"中明确指出偏移点。
- 超目的实现（方案有、requirements.md 无）→ 校验者 1 §2.3 按"无主设计 / 超目的实现"处理。

#### 1.1 五块上下文收集

按 [@references/00-context-template.md](references/00-context-template.md) 收集业务 / 技术 / 现有代码库与既有约定 / 系统规模 / SLA 五块。方案中已写明的直接抽取并向用户复核；`requirements.md` 中已写明但方案未抽取的，**优先采用 requirements.md 的值**并标注来源；两者均缺失项标 "未说明"。

**关键 4 项缺项处理**（业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标）：
- 正式评审：任一缺失必须追问后再继续；如 `requirements.md` 已明确给出，视为不缺。
- 初步扫描：可临时假设并继续，但必须在报告头部列出假设与置信度。

### Step 1.5 — 外部假设核查（防"基础假设崩塌"）

> 实现阶段推翻方案，往往不是"方案内部不自洽"，而是"方案对外部现实（仓库 / 中间件部署形态 / 既有约定）的假设不成立"。本步用证据驱动的方式提前暴露这类风险。

**触发信号（任一即必须执行；都不具备可跳过，但需在报告中说明跳过原因）**：

- 方案首次引入仓库 / 团队从未使用过的中间件、命令族、能力或机制。
- 方案要替换或扩展一个既有机制（核对"现在怎么做、为什么要换"）。
- 方案关键路径依赖某外部能力提供特定语义（如"可靠投递 / 互斥 / 强一致 / 零丢失 / 全局有序"）。

**必做项（凭证据，不凭记忆）**：

1. **既有约定核查**：在仓库代码 / 注释 / 架构规范中检索方案使用的关键能力 / 命令 / 类型名，判断：是否首次引入（命中数为 0）/ 是否已被标记为 `deprecated` / `do not use` / `unreliable` / 仓库是否已有同类机制。
2. **能力假设对照**：把方案中所有"我假设 X 能做到 Y"的论断显式列出，逐条与该依赖在**本系统实际部署形态**下的真实能力比对（不是官方文档的最大能力，而是本系统配置下能拿到的能力）。
3. **替代 / 兜底盘点**：若首次引入或与既有机制冲突，盘点是否有不依赖新机制的等价方案，以及语义不满足时的补偿路径（对账 / reconcile / 全量重同步）。

核查结果在最终报告"外部假设核查"小节逐项记录：检查项 / 证据来源（含检索命令）/ 与方案假设是否一致 / 冲突点。

**冲突处置**：

- 与既有约定不一致 → Step 4 由一致性校验者按其"方案 vs 外部现实一致性"清单输出问题（[@references/01-consistency-completeness.md](references/01-consistency-completeness.md)）。
- 能力假设不成立且无兜底 → Step 4 由 HA 校验者按其"跨节点协调通道可靠性"清单输出问题（[@references/02-fault-tolerance-ha.md](references/02-fault-tolerance-ha.md)），Step 6 按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第三节"外部假设错位"规则强制升级为 Blocker。

### Step 1.6 — 仓库规则与禁用事项预扫描（防"违反明确规定的不允许事项"）

> 目的：回答"**技术方案是否违反了当前仓库里面明确规定的一些不允许事项 / 弃用的公共方法**"。一次扫描，多个校验者复用，避免脑补。

**做法**：按 [@references/11-repo-rules-and-bans.md](references/11-repo-rules-and-bans.md) 的检索范围与产出格式执行：

1. 检索仓库规则文件（cursor rules / coding standards / 安全规范 / SECURITY.md / 架构文档 / 历史复盘 / linter 配置 / CI 拦截规则 等）。
2. 检索代码注释中的 `deprecated` / `do not use` / `禁用` / `已废弃` / `unsafe` 等显式标记。
3. 把所有命中条目归并为**禁用清单**（`BAN-1`、`BAN-2`、…），含：类别 / 条目 / 来源（文件 + 行号）/ 适用范围 / 严重度 / 是否可豁免。
4. 把禁用清单显式列在最终报告"仓库规则与禁用项预扫描"小节，作为后续校验者的对账锚点。

**空清单处置**：

- 仓库内**无**任何书面禁用规定 → 必须在产物中显式记录"未检索到仓库规则文件 / 禁用清单为空"，**不得静默跳过**。
- 不得用"通用经验"代替"仓库书面规则"作为禁用判定依据；通用经验只能作为风险提示。

**注入下游**：禁用清单产出后，将其作为上下文注入校验者 1（§12 仓库规则与禁用事项一致性）、校验者 4（安全相关禁用项）、校验者 5（数据 / 字段类型相关禁用项）、校验者 6（运维 / 日志 / 配置相关禁用项）、校验者 8（deprecated 公共方法）。所有命中项最终在报告"仓库规则违反清单"小节集中呈现。

### Step 2 — 识别方案类型（决定校验重点）

根据方案内容判定属于以下哪一种（可多选；跨类型取并集）。校验者编号详见 Step 4 表，下表使用编号引用。

| 方案类型 | 默认强制启用的校验者 | 强化重点 |
|---|---|---|
| 新系统建设 | 1 + 2 + 3 + 5 + 6 + 7 | 端到端完整性、容量评估、运维准入 |
| 老系统改造 | 1 + 6 + 7 + 8 | 兼容性、灰度、回滚、数据迁移 |
| 数据迁移 | 1 + 5 + 6 + 8 | 双写 / 影子读 / 对账、迁移期一致性、回滚 |
| API / 接口设计 | 1 + 4 + 7 + 8 | 契约、幂等、错误码、版本、鉴权 |
| 高并发活动 | 1 + 2 + 3 + 6 + 7 | 容量、限流降级、热点、预案 |
| 支付 / 交易 / 账务 | 全部启用 + 严格模式 | 资金一致性、对账、合规、审计、零丢失 |
| 消息异步化 | 1 + 2 + 5 + 6 + 8 | 可靠投递、幂等消费、积压、协议演进 |
| 存储选型 | 1 + 3 + 5 + 7 | 存储匹配度、数据建模、扩展上限 |
| 容灾改造 | 2 + 5 + 6 + 7 | RPO / RTO、切换、脑裂、回滚 |
| 权限 / 安全改造 | 1 + 4 + 6 | 越权、审计、合规、灰度 |
| 降本 / 性能优化 | 3 + 6 + 7 | 瓶颈定位、量化收益、回滚兜底 |
| 内部工具 / MVP | 1（轻量）+ 7 | 正确性 + 防过度设计 |

识别结果先告诉用户。

### Step 3 — 选择校验者组合

以 Step 2 矩阵为底，按业务重要性调档：

| 重要性 | 深度 |
|---|---|
| 核心系统 | 全部 8 个，严格模式 |
| 重要系统 | 必选组（1 + 7）+ 3~4 个按需 |
| 一般系统 | 必选组（1 + 7）+ 1~2 个按需 |
| 内部工具 / MVP | 仅 1（轻量）+ 7 |

把"本次启用了哪几个校验者、为什么"先告诉用户，允许增减。范围限定评审（如"只看性能"）下仅展开指定校验者；若发现明显跨维度 Blocker，必须在报告"额外风险提示"小节简述并建议补充评审范围。

### Step 4 — 逐个校验者执行

对选中的每一个校验者：

1. 读取对应 reference 文件（见下表），完全采用其中定义的 Role / Identity / 审查清单 / Rules——这是该校验者的 system prompt，不要改写或缩水。
2. **Output Format 统一使用 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第七节模板**（位置 / 原文证据 / 判断依据 / 修改建议 + 维度专属字段），编号前缀按下表。
3. 把 Step 1 收集的上下文 + Step 1.0 预读的 `requirements.md`（如存在）+ Step 1.5 外部假设核查结论 + Step 1.6 禁用清单注入当前校验者的视角，逐项过审查清单。
4. **强制输出的对账矩阵**（缺一不可，否则该校验者视为未交付）：
   - 校验者 1：**功能完整性矩阵**（FR → 设计落点；FR 来源 = `requirements.md`（如存在，**优先**）∪ 方案内需求章节，对每条 FR 标注"FR 来源"列）+ **NFR 满足度矩阵**（NFR → 支撑设计）+ **关键逻辑详细度统计** + **仓库禁用项命中清单**（消费 Step 1.6 产出）。
   - 校验者 2：**可用性 NFR 满足度矩阵**。
   - 校验者 3：**性能 NFR 满足度矩阵**。
   - 校验者 4：**安全 NFR 满足度矩阵**。
   - 校验者 5：**关键存储与关键数据结构齐备度矩阵**。
5. 给出该维度的**总结 + 评分（1–10，按 09 第六节 Rubric）**。

|---|---|---|
| 1 | 一致性与完整性（含功能完整性 / NFR 对账 / 关键逻辑详细度 / 仓库禁用项一致性） | [@references/01-consistency-completeness.md](references/01-consistency-completeness.md) |

**所有发现强制带证据**：每个问题必须包含位置、原文证据（缺失则写"方案未说明"）、判断依据、修改建议。这是合格交付的硬要求，参见 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md)。

### Step 5 — 汇总仲裁

按 [@references/08-orchestration.md](references/08-orchestration.md) 第二节的优先级排序、量化判断与不可妥协红线做最终裁决。一句话原则：**正确性 > 安全 > 数据 > 需求驱动的健壮性 > 兼容性 > 简洁性 > 非需求驱动的健壮性 / 性能**。

### Step 6 — 严重程度归一化与门禁判定

汇总所有问题前，必须执行：

1. **归一化**：按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第二节的映射表，把各校验者原始等级（Critical/P0/High 等）统一映射到 5 档（Blocker/High/Medium/Low/Info）。
2. **Blocker 自动判定**：按 09-severity-and-gate.md 第三节规则，把符合条件的问题强制升级为 Blocker。
3. **门禁判定**：按 09-severity-and-gate.md 第四节规则，机械推导"是否可进入下一阶段"。**不允许凭主观判断给结论**。
4. **豁免登记**：High 及以下问题如不修复，必须按 09-severity-and-gate.md 第五节登记 Waiver；Blocker 不可豁免。

> 初步扫描模式跳过本步的"门禁判定"，仅做归一化和风险列举。

### Step 7 — 输出最终报告

#### 7.1 产物落盘规则（强约束）

> **本步是动作指令，不是规则描述**。Agent 必须**真正调用文件写入工具**把报告内容写到磁盘上的目标文件；只在对话里贴出报告内容**不算**完成本步。本步未完成时，Step 7 视为失败，整个评审视为未交付。

**统一产物根目录（强制）**：本 skill 的**所有产物**——最终评审报告、上一版备份、所有中间产物（外部假设核查记录、仓库规则与禁用项预扫描清单、对账矩阵草稿等）——一律落盘到 **`TECH_DESIGN_DIR`**（解析规则见下，固定在原仓库根目录），**不得**把任何产物散落到原方案目录或其它位置。

- 与 `qiq-backend-tech` 技术方案生成 skill 配对：spec skill 也以仓库根 `.tech-design/` 为产物根目录，本 skill 沿用同一目录，形成"方案产出 ↔ 评审产物"的同址闭环。
- 该目录如不存在需先**创建**（一次真实文件系统动作），再写入产物。
- 若原方案不是来自具体文件（如用户直接粘贴文本），先询问用户期望的工作目录；未指定时默认在**当前工作目录**下创建并使用 `.tech-design/`。

#### 7.1.0 `TECH_DESIGN_DIR` 解析规则（**读 / 写共享同一规则，全文唯一权威定义**）

> 关键约定：`.tech-design/` **固定**位于**原仓库根目录**下（项目级，多份方案共享一个目录），**不**位于方案文件所在子目录。读（Step 1.0 预读 `requirements.md` / 历史中间产物）和写（Step 7 落盘）使用同一套解析结果，避免读一处、写另一处。

**解析步骤**（必做，按顺序执行）：

1. **定位 `REPO_ROOT`**：从原方案文件所在目录开始，逐级向上回溯，遇到第一个含 `.git` / `.hg` / `.svn` 任一标识的目录即为 `REPO_ROOT`；回溯到磁盘根仍未命中则视为"无仓库根"。粘贴文本场景：以当前工作目录作为起点。
2. **取仓库根 `.tech-design/`**：若 `REPO_ROOT` 存在 → `TECH_DESIGN_DIR = REPO_ROOT/.tech-design`（不论该目录当前是否存在；不存在则在写场景下真实创建）。
3. **无仓库根的兜底**：若步骤 1 未找到 `REPO_ROOT`（散文件 / 粘贴文本 / 临时目录场景）→ 先询问用户期望工作目录；未指定时取**当前工作目录**作为根，`TECH_DESIGN_DIR = <CWD>/.tech-design`。
4. **不在方案目录建产物**：即使方案文件所在子目录下已存在历史的 `.tech-design/`，本 skill 也**不**采用该子目录；统一以仓库根为准。如发现这种历史散落，可在最终报告"备注"中提示用户清理，但**不**改变本次 `TECH_DESIGN_DIR` 解析结果。

**读 / 写一致性硬约束**：

- Step 1.0 读取 `requirements.md` / 历史中间产物，**必须**走同一套解析，把命中的 `TECH_DESIGN_DIR` 缓存为本次评审的全局变量；后续 Step 7 落盘时**直接复用**，不得重新解析到不同路径。
- 一次评审过程中 `TECH_DESIGN_DIR` 解析一次，全程不变。

**目录布局**（以解析后的 `TECH_DESIGN_DIR` 为根；父目录恒为仓库根）：

```
<REPO_ROOT>/                                       # 原仓库根目录（含 .git/.hg/.svn 标识）
├── <方案路径>/<原方案文件>.md                      # 原方案所在位置（可能在仓库任意子目录）
└── .tech-design/                                   # 本 skill 唯一产物根目录（= TECH_DESIGN_DIR）
    ├── requirements.md                             # （由 spec skill 产出；本 skill 只读不写）
    ├── <原文件名>-review.md                        # 最终评审报告（稳定路径，每次重跑覆盖写）
    ├── <原文件名>-review.bak.YYYYMMDD-HHmmss.md    # 上一版备份（重跑时由当前最终报告重命名而来）
    └── intermediates/                              # 中间产物子目录（可选，但建议产出）
        ├── <原文件名>-context.md                   # Step 1 上下文收集摘要
        ├── <原文件名>-assumption-check.md          # Step 1.5 外部假设核查记录
        └── <原文件名>-repo-bans.md                 # Step 1.6 仓库规则与禁用项预扫描清单
```

**硬动作（必做，按顺序执行，不可省略）**：

1. 按 §7.1.0 解析得到 `TECH_DESIGN_DIR`；若不存在则真实创建。
2. 计算目标落盘路径 `REPORT_PATH = <TECH_DESIGN_DIR>/<原方案文件名（不含扩展名）>-review.md`。
3. 若 `REPORT_PATH` 已存在，先按下文"重跑策略"将旧文件**重命名为备份**（也是一次真实的文件系统动作，不能只在对话里说"我会备份"）。
4. **调用可写入文件的工具**（如 `edit_file` / `write` / `replace_in_file` 等当前可用的写文件工具，取决于运行环境），把**完整的最终报告 Markdown 内容**写入 `REPORT_PATH`。
5. 在对话回复中**显式回显**：`REPO_ROOT` 绝对路径（无仓库根场景填"无（兜底使用当前工作目录 <CWD>）"）、`TECH_DESIGN_DIR` 绝对路径、`REPORT_PATH` 绝对路径、上一版备份绝对路径（无则写"无"）。这是给用户和 Verification 自检的可验证凭据。
6. **禁止替代行为**：不得用"我已生成报告（在下面）" / "完整报告见上方/下方" / 把整份报告贴在对话里 等方式替代真实落盘。如果当前环境**确实**没有可写文件的工具，必须**显式告知用户"无法落盘，仅以对话形式输出"**，并在 Verification 自检中标记为未达成；不得静默降级。
7. **禁止跨目录落盘**：报告路径必须以**仓库根**下的 `.tech-design/` 作为最后一级父目录；落盘到原方案文件所在子目录、当前工作目录根（有仓库根时）、临时目录等均视为本步失败。

**命名规则**：

- 报告：`<TECH_DESIGN_DIR>/<原方案文件名（不含扩展名）>-review.md`
- 备份：`<TECH_DESIGN_DIR>/<原方案文件名（不含扩展名）>-review.bak.YYYYMMDD-HHmmss.md`
- 中间产物（如产出）：`<TECH_DESIGN_DIR>/intermediates/<原方案文件名（不含扩展名）>-<intermediate-kind>.md`

例：

- 仓库根 `/repo/`，方案 `/repo/docs/order-system.md` → `TECH_DESIGN_DIR = /repo/.tech-design`，报告 `/repo/.tech-design/order-system-review.md`（**不**落到 `/repo/docs/.tech-design/`）。
- 仓库根 `/repo/`，方案 `/repo/RFC-2025-001.md`（在仓库根） → `TECH_DESIGN_DIR = /repo/.tech-design`，报告 `/repo/.tech-design/RFC-2025-001-review.md`。
- 方案 `/path/to/RFC-2025-001.md`，向上回溯**未命中**任何 `.git`/`.hg`/`.svn` → 无仓库根，按兜底走当前工作目录；若 `CWD = /workspace`，则 `TECH_DESIGN_DIR = /workspace/.tech-design`，报告 `/workspace/.tech-design/RFC-2025-001-review.md`。
- 粘贴文本未指定路径时：`CWD/.tech-design/tech-review-YYYYMMDD-HHmmss.md`，并在头部"原方案路径"字段标注"用户直接粘贴文本"。

**重跑策略（备份 + 覆盖写）**：若同名 `<原文件名>-review.md` 已在 `.tech-design/` 中存在：

1. 先把旧文件重命名为 `<原文件名>-review.bak.YYYYMMDD-HHmmss.md`（时间戳为本次重跑时刻，到秒），作为备份保留在**同一 `.tech-design/` 目录**内。
2. 以本次报告**覆盖写**原文件名 `<原文件名>-review.md`，保证"最新一次评审结果始终位于同一个稳定路径"。
3. 将备份产生的路径写到报告头部 `上一版备份路径` 字段。
4. **不追问**、**不拒写**、**不产生多份报告**。用户仅在明确表达"不需要备份"时才可跳过第 1/3 步。
5. 中间产物（intermediates/）每次重跑同样**覆盖写**同名文件，不再额外备份；如确需保留，由用户显式要求。

> 备份文件名示例：`.tech-design/order-system-review.md` → `.tech-design/order-system-review.bak.20260521-215708.md`。

**中间产物（建议产出，强烈推荐）**：以下中间产物如已实际产出（如 Step 1.5 / Step 1.6 真的执行过），应同步落盘到 `intermediates/` 子目录，便于审计与重放：

- Step 1 上下文收集摘要 → `<原文件名>-context.md`
- Step 1.5 外部假设核查记录 → `<原文件名>-assumption-check.md`
- Step 1.6 仓库规则与禁用项预扫描清单 → `<原文件名>-repo-bans.md`

中间产物在最终报告对应小节中以**摘要 + 链接**方式引用，避免最终报告膨胀；如未产出独立中间产物文件，最终报告对应小节必须自包含完整内容。

**版本号写入**：报告头部必须记录本次评审使用的 skill 版本号（取自本文件 frontmatter 的 `version` 字段，以 SKILL.md 为唯一权威来源），以便追溯"这份报告是哪个版本的 skill 跑出来的"。升级 skill 后重跑会自动使用新版本号。

**报告头部必须显式包含的元信息**：

- 原方案路径
- **原仓库根目录**（`REPO_ROOT` 绝对路径；无仓库根时填"无（兜底使用当前工作目录 `<CWD>`）"）
- **原始需求记录路径**（`<TECH_DESIGN_DIR>/requirements.md` 的绝对路径；不存在填"无（未发现 .tech-design/requirements.md）"）
- 产物根目录（`TECH_DESIGN_DIR` 绝对路径，恒为 `<REPO_ROOT>/.tech-design/` 或兜底工作目录下的 `.tech-design/`）
- 报告输出路径（绝对路径，必须位于 `.tech-design/` 内）
- 上一版备份路径（首次跑填"无"；重跑填备份文件绝对路径，必须位于 `.tech-design/` 内）
- 中间产物清单（已落盘的中间产物文件名列表；未产出填"无"）
- 评审时间（本次跑的时间，到秒）
- skill 版本号

#### 7.2 报告内容结构

按 [@templates/final-report.md](templates/final-report.md) 组织，含一个执行摘要 + 11 个正文章节（与 final-report.md 章节编号一一对应）：

4. **对账矩阵汇总**（强制章节，所有校验者矩阵的集中呈现）：
   - 4.1 功能完整性矩阵（FR → 设计落点）
   - 4.2 NFR 满足度矩阵（含可用性 / 性能 / 安全 / 可观测性 4 项强制 + 其它声明的 NFR）
   - 4.3 关键存储与关键数据结构齐备度矩阵
   - 4.4 关键逻辑详细度统计
   - 4.5 仓库规则违反清单（命中禁用清单的方案位置汇总，含每条命中的"BAN- 编号 / 方案位置 / 原文证据 / 严重度"）

6. **冲突仲裁结论**（如果不同维度建议冲突）。
7. **整改清单**（按 Blocker / High / Medium / Low / Info 分组的字段化表格：编号 / 整改项 / Owner / 验收标准 / 是否阻塞 / 建议完成时间）。
8. **豁免登记表**（仅 High 及以下可豁免，Blocker 不可豁免；安全 / 资金一致性 / 数据丢失 / 合规类不允许豁免）。

## Pitfalls

下列是高频翻车点的反向描述。**详细的"应该怎么做对"以正向 checklist 形式收敛在 Verification 自检清单中**，跑完后逐条勾选即可。

- **跑全部校验者** → 上下文爆炸 + 噪声淹没用户。按方案类型 + 特征裁剪，并解释为何裁剪。
- **冲突时不仲裁** → 用户拿到矛盾建议。Step 5 必须给出明确取舍。
- **凭印象写问题 / 不引用方案原文** → 必须引用原文作为证据；方案未说明的明确写出。
- **凭外部依赖的"理论能力"评审** → 论文上能做到，本系统部署形态下不一定能做到。出现 Step 1.5 三个信号任一项时必须走 Step 1.5。
- **跳过仓库规则预扫描** → 拍脑袋说"违反了 XX 规范"或漏掉真实存在的禁用条目。Step 1.6 必须执行；空清单必须显式声明，不得静默跳过。
- **漏读原始需求记录（`requirements.md`）** → 评审只看方案，看不到方案对原始诉求的"漏抽 / 改写 / 静默删减"。Step 1.0 必须先按 §7.1.0 规则解析 `TECH_DESIGN_DIR`（**固定 = `<REPO_ROOT>/.tech-design/`**），再检测 `${TECH_DESIGN_DIR}/requirements.md` 是否存在；存在就**真读**，并把它作为 FR 矩阵源头之一（优先级 ≥ 方案内需求章节）；不存在显式记录"无"，不得静默跳过。
- **`TECH_DESIGN_DIR` 落地漂移** → 把产物落到原方案文件所在子目录（如 `/repo/docs/.tech-design/`）而非仓库根（`/repo/.tech-design/`），导致与 spec skill 失去同址闭环、同一仓库下多份方案产物散落难治理。本 skill 的 `TECH_DESIGN_DIR` **恒**为 `<REPO_ROOT>/.tech-design/`（无仓库根时退到 `<CWD>/.tech-design/`），一次评审解析一次、读写共用，严禁就地用方案目录子级 `.tech-design/`。
- **只列问题不出对账矩阵** → 用户拿不到"功能是否完整 / NFR 是否达标 / 关键存储是否齐备"的全景判断。Step 4 第 4 点列出的 5 类矩阵缺一不可。
- **跳过上下文、输出格式漂移、只列问题不给建议、严重程度跨维度漂移、门禁凭感觉、只挑刺不肯定** → 在 Verification 自检中逐条收敛。
- **Step 7 落盘四类违规**：①只在对话里贴报告不调用写工具 ②重跑不真备份就覆盖 ③报告头不写 skill 版本号 ④把产物落盘到原方案目录、原方案文件所在子目录的 `.tech-design/`、当前工作目录根（有仓库根时）或其它位置而非仓库根的 `.tech-design/` → 用户拿不到稳定路径产物 / 重跑无从 diff / 旧报告无法追溯 / 产物散落难治理。Step 7.1 的硬动作必须**真的执行**，所有产物（报告 + 备份 + 中间产物）必须统一落到 §7.1.0 解析出的 `TECH_DESIGN_DIR`（恒为 `<REPO_ROOT>/.tech-design/`），并在对话里回显 `REPO_ROOT` / `TECH_DESIGN_DIR` / `REPORT_PATH` 三个绝对路径。

### 反例与正例

**❌ 不合格的发现**：
> 性能可能有问题，建议优化。

**✅ 合格的发现**：
> ### [PERF-3] [瓶颈] [严重程度: High]
> **位置**：方案 §4.2 订单查询接口
> **原文证据**："订单查询通过 SELECT * FROM orders WHERE user_id=? ORDER BY create_time DESC"
> **判断依据**：用户规模 1000 万，单用户历史订单可达数千条；订单表 1 年后预估 5 亿行
> **问题描述**：未限定时间范围 + 无分页 + ORDER BY 非索引前缀，深度查询将导致 P99 飙升至秒级
> **修改建议**：1) 强制时间范围（默认近 90 天）；2) 游标分页；3) (user_id, create_time) 联合索引

**❌ 不合格的"功能完整性"判定**：
> FR 都基本覆盖了。

**✅ 合格的"功能完整性"判定**：
> 功能完整性矩阵：FR-001 / FR-002 / FR-003 完整度 ✅；FR-004 完整度 🟡（核心路径有，但缺超时/重试设计）；FR-005 完整度 ❌（仅在需求章节出现，详细设计与流程章节均无对应）。Blocker 升级：FR-005 为 P0 级且完整度 ❌ → C-Critical → Blocker。
