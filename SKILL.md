# qiq-backend-tech-review — 互联网后台技术方案多维校验

## When to use

满足任一条件就启用：

- 用户给出一份后台/分布式/微服务/数据系统/高并发系统的技术方案/设计文档/RFC，希望做"评审"、"review"、"挑刺"、"找漏洞"、"上线前检查"。
- 用户提到关键词：**技术方案评审 / 架构评审 / 设计评审 / design review / RFC review / 上线前评审 / 技术方案校验**。
- 用户希望对一份方案做 **多维度系统化检查**，而不是单一关注点提问（如只问性能）。

不适用：

- 仅纠结于一个具体技术问题（如"Redis 为什么慢"）——直接回答即可。
- 前端/移动端/算法模型方案——本 skill 维度不匹配。
- 业务需求评审、PRD 评审——本 skill 关注技术实现方案。

## 执行模式（先确认模式再开始）

启动时必须先与用户对齐执行模式，二选一：

### 正式评审模式（默认）
- 严格执行 Step 1 上下文收集，缺关键 4 项（业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标）必须**追问后再开始**。
- 输出包含完整的"是否可进入下一阶段"门禁结论。
- 适合：上线前评审、正式 design review、RFC 评审。

### 初步扫描模式
- 用户明确表达"先粗看 / 快速过一遍 / 不用追问 / 不需要门禁"时启用。
- 允许在上下文不全的情况下先扫一遍，**但报告头部必须声明**：
  - 上下文置信度（高/中/低）
  - 临时采用的关键假设（逐条列出）
  - 哪些结论必须在补充上下文后复核
- **不输出**最终"是否可上线"的门禁结论，仅输出"当前可见的风险清单"。
- 适合：初稿自检、需求方快速对齐、方案早期讨论。

> 用户没明确说时默认用"正式评审模式"，并把模式选择告诉用户、允许切换。

## Workflow

### Step 1 — 收集审查上下文（必须）

**禁止跳过此步直接 review**。先把上下文拉齐，否则校验深度会失真（核心系统和内部工具不能用同一把尺子）。

按 [@references/00-context-template.md](references/00-context-template.md) 收集：

- **业务**：重要性等级（核心/重要/一般）、业务领域、用户规模（DAU/MAU）、是否涉及资金。
- **技术**：团队规模、团队熟悉的技术栈、现有基础设施、上线时间窗。
- **规模**：当前 / 1 年后预期 QPS、当前 / 1 年后预期数据量。
- **SLA**：可用性目标、核心接口延迟（P99）、一致性要求、RPO、RTO、部署模式（单机房 / 同城双活 / 异地多活）。

如果方案中已写明，直接抽取并向用户复核；缺失项标注"未说明"。

**关键缺项处理**：
- 正式评审模式：缺"业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标"任一项 → 必须追问后再继续。
- 初步扫描模式：缺项可临时假设并继续，但必须在报告头部明确列出假设与置信度。

### Step 2 — 识别方案类型（决定校验重点）

根据方案内容判定属于以下哪一种（可多选）：

| 方案类型 | 默认强制启用的校验者 | 强化重点 |
|---|---|---|
| 新系统建设 | 1 + 2 + 3 + 5 + 6 + 7 | 端到端完整性、容量评估、运维准入 |
| 老系统改造 | 1 + 6 + 8 + 7 | 兼容性、灰度、回滚、数据迁移 |
| 数据迁移 | 1 + 5 + 6 + 8 | 双写/影子读/对账、迁移期一致性、回滚 |
| API / 接口设计 | 1 + 4 + 8 + 7 | 契约、幂等、错误码、版本、鉴权 |
| 高并发活动 | 1 + 2 + 3 + 6 + 7 | 容量、限流降级、热点、预案 |
| 支付 / 交易 / 账务 | 全部启用 + 严格模式 | 资金一致性、对账、合规、审计、零丢失 |
| 消息异步化 | 1 + 2 + 5 + 6 + 8 | 可靠投递、幂等消费、积压、协议演进 |
| 存储选型 | 1 + 3 + 5 + 7 | 存储匹配度、数据建模、扩展上限 |
| 容灾改造 | 2 + 5 + 6 + 7 | RPO/RTO、切换、脑裂、回滚 |
| 权限 / 安全改造 | 1 + 4 + 6 | 越权、审计、合规、灰度 |
| 降本 / 性能优化 | 3 + 7 + 6 | 瓶颈定位、量化收益、回滚兜底 |
| 内部工具 / MVP | 1（轻量）+ 7 | 仅做正确性 + 防过度设计 |

如果方案跨多个类型，取并集。把识别结果先告诉用户。

### Step 3 — 选择校验者组合

以 Step 2 矩阵为权威基础，按业务重要性调档；详见 [@references/08-orchestration.md](references/08-orchestration.md) 的"分组策略"。

**必选组（每个方案都跑）**：校验者 1（一致性）+ 校验者 7（简洁性）。

**按业务重要性调档**

| 重要性 | 深度 |
|---|---|
| 核心系统 | 全部 8 个，严格模式 |
| 重要系统 | 必选组 + 3~4 个按需 |
| 一般系统 | 必选组 + 1~2 个按需 |
| 内部工具 / MVP | 仅校验者 1（轻量）+ 校验者 7 |

把"本次启用了哪几个校验者、为什么"先告诉用户，让他可以增减。

**范围限定评审**：
用户可指定"只审某一部分"（如"只看性能"）。此时仅展开指定范围的校验者；若发现明显跨维度高风险（特别是 Blocker），必须在报告"额外风险提示"小节简述并建议补充评审范围。

### Step 4 — 逐个校验者执行

对选中的每一个校验者：

1. 读取对应 reference 文件（见下表）。
2. **完全采用该 reference 中定义的 Role / Identity / 审查清单 / Output Format / Rules**——这是该校验者的 system prompt，不要自己改写或缩水。
3. 把 Step 1 收集的上下文注入到当前校验者的视角。
4. 逐项过审查清单，凡发现问题，按该校验者规定的 Output Format 输出。
5. 给出该维度的 **总结 + 评分（1–10，按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) Rubric）**。

| # | 校验者 | Reference |
|---|---|---|
| 1 | 一致性与完整性 | [@references/01-consistency-completeness.md](references/01-consistency-completeness.md) |
| 2 | 容错与高可用 | [@references/02-fault-tolerance-ha.md](references/02-fault-tolerance-ha.md) |
| 3 | 性能与可扩展性 | [@references/03-performance-scalability.md](references/03-performance-scalability.md) |
| 4 | 安全性 | [@references/04-security.md](references/04-security.md) |
| 5 | 数据一致性与存储 | [@references/05-data-storage.md](references/05-data-storage.md) |
| 6 | 可运维性 | [@references/06-operability.md](references/06-operability.md) |
| 7 | 简洁性（奥卡姆剃刀） | [@references/07-simplicity.md](references/07-simplicity.md) |
| 8 | API 契约与兼容性 | [@references/10-api-contract-compatibility.md](references/10-api-contract-compatibility.md) |

**所有发现强制带证据**：每个问题必须包含位置、原文证据（缺失则写"方案未说明"）、判断依据、修改建议。这是合格交付的硬要求，参见 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md)。

### Step 5 — 汇总仲裁

按 [@references/08-orchestration.md](references/08-orchestration.md) 的"冲突仲裁规则"做最终裁决：

**优先级排序**

1. 正确性（1）> 一切——逻辑矛盾/遗漏必须修，不论复杂度。
2. 安全性（4）> 简洁性——安全漏洞必须修。
3. 数据正确性（5）> 性能 > 简洁性——涉及钱的数据一致性不让步。
4. 需求驱动的健壮性（2）> 简洁性（7）——SLA 明确要求的容错不能砍。
5. 兼容性（8）> 性能 / 简洁性——破坏性变更必须先解决。
6. 简洁性（7）> 非需求驱动的健壮/性能——非 SLA 要求的过度设计要砍。

**量化判断**

- 故障概率 × 影响 > 长期维护成本 → 采纳健壮性建议。
- 故障概率 × 影响 < 长期维护成本 → 采纳简洁性建议。

### Step 6 — 严重程度归一化与门禁判定

汇总所有问题前，必须执行：

1. **归一化**：按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第二节的映射表，把各校验者原始等级（Critical/P0/High 等）统一映射到 5 档（Blocker/High/Medium/Low/Info）。
2. **Blocker 自动判定**：按 09-severity-and-gate.md 第三节规则，把符合条件的问题强制升级为 Blocker。
3. **门禁判定**：按 09-severity-and-gate.md 第四节规则，机械推导"是否可进入下一阶段"。**不允许凭主观判断给结论**。
4. **豁免登记**：High 及以下问题如不修复，必须按 09-severity-and-gate.md 第五节登记 Waiver；Blocker 不可豁免。

> 初步扫描模式跳过本步的"门禁判定"，仅做归一化和风险列举。

### Step 7 — 输出最终报告

按 [@templates/final-report.md](templates/final-report.md) 组织最终交付：

1. **执行摘要**（Top 5 风险 + 是否可进入下一阶段 + 最大阻塞项 + 需业务方确认事项）。
2. **审查上下文摘要**（Step 1 收集的） + **假设与置信度**（初步扫描模式必填）。
3. **方案类型识别 + 本次启用的校验者列表 + 启用原因**。
4. **各校验者的发现**（按维度组织，每个问题保留原始编号如 `C-1`、`HA-3`、`SEC-2`、`API-1` 便于追溯，同时标注归一化后的统一严重程度）。
5. **冲突仲裁结论**（如果不同维度建议冲突）。
6. **整改清单**（字段化表格：编号 / 整改项 / Owner / 验收标准 / 是否阻塞 / 建议完成时间）。
7. **豁免登记表**（如有）。
8. **设计亮点**（无需调整 / 无需进一步优化的设计）。
9. **整体评分汇总表 + 门禁判定结论**（按 09-severity-and-gate.md 推导，不允许"看情况"）。

## Pitfalls

- **跑全部校验者** → 上下文爆炸 + 用户淹没在噪声里。按方案类型 + 特征裁剪，并向用户解释为何裁剪。
- **冲突时不仲裁** → 用户拿到的是矛盾的建议。Step 5 必须给出明确取舍。
- **凭印象写问题** → 必须"引用方案原文作为证据"，不能脑补；方案未说明的明确写出。

> 其它常见反向问题（跳过上下文、输出格式漂移、只列问题不给建议、严重程度跨维度漂移、门禁凭感觉、只挑刺不肯定）均以正向 checklist 形式收敛在下方 Verification 自检清单中，跑完后逐条勾选即可。

### 反例与正例

**❌ 不合格的发现**：
> 性能可能有问题，建议优化。

**✅ 合格的发现**：
> ### [PERF-3] [瓶颈] [严重程度: High]
> **位置**：方案 §4.2 订单查询接口
> **原文证据**："订单查询通过 SELECT * FROM orders WHERE user_id=? ORDER BY create_time DESC"
> **判断依据**：用户规模 1000 万，单用户历史订单可达数千条；订单表 1 年后预估 5 亿行
> **问题描述**：未限定时间范围 + 无分页 + ORDER BY 非索引前缀，深度查询将导致 P99 飙升至秒级
> **修改建议**：1) 强制时间范围（默认近 90 天）；2) 游标分页；3) (user_id, create_time) 联合索引

## Verification

跑完后自检以下条目，全部满足才算合格交付：

- [ ] 已声明执行模式（正式评审 / 初步扫描）。
- [ ] 已记录业务重要性 / 是否涉及资金 / QPS / SLA 这 4 项核心上下文（初步扫描模式可为假设）。
- [ ] 已识别方案类型，并说明启用了哪几个校验者、为何。
- [ ] 每个发现都包含：位置、原文证据（或"方案未说明"）、判断依据、统一严重程度、修改建议。
- [ ] 简洁性建议都包含：替代方案、前提条件、失效条件。
- [ ] 严重程度已按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 归一化为 Blocker/High/Medium/Low/Info。
- [ ] 已执行 Blocker 自动判定规则（资金/数据丢失/安全/合规/SLA 错位等）。
- [ ] 已给出字段化的整改清单（含 Owner / 验收标准 / 是否阻塞）。
- [ ] 如有豁免，已登记 Waiver（含 owner / 补救方案 / 复盘时间）。
- [ ] 已给出"设计亮点"小节。
- [ ] 正式评审模式下，已按门禁规则给出"允许 / 不允许进入下一阶段"的明确结论；初步扫描模式下，已声明置信度并未输出门禁结论。
