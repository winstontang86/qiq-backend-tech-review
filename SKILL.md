# qiq-backend-tech-review — 互联网后台技术方案多维校验

## When to use

启用条件（任一即可）：

- 用户给出后台 / 分布式 / 微服务 / 数据系统 / 高并发系统的技术方案 / 设计文档 / RFC，希望做"评审 / review / 挑刺 / 找漏洞 / 上线前检查"。
- 用户希望对一份方案做**多维度系统化检查**，而非单点提问（如只问性能）。

不适用：

- 单点技术问题（如"Redis 为什么慢"）——直接回答即可。
- 前端 / 移动端 / 算法模型方案——本 skill 维度不匹配。
- 业务需求 / PRD 评审——本 skill 关注技术实现方案。

## 执行模式（先与用户对齐再开始）

二选一，用户未明确说明时默认"正式评审"，并把当前模式告诉用户、允许切换：

| 模式 | 触发 | 上下文要求 | 是否输出门禁结论 | 适合场景 |
|---|---|---|---|---|
| **正式评审**（默认） | 默认；用户希望严谨 review | 缺关键 4 项必须追问后再开始（业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标） | 是 | 上线前评审、正式 design review、RFC 评审 |
| **初步扫描** | 用户明确表达"先粗看 / 快速过一遍 / 不用追问" | 允许临时假设继续，但报告头部必须声明置信度、关键假设、需复核结论 | 否，仅输出风险清单 | 初稿自检、早期讨论、需求方快速对齐 |

## Workflow

### Step 1 — 收集审查上下文（必须）

**禁止跳过此步直接 review**：核心系统和内部工具不能用同一把尺子。

按 [@references/00-context-template.md](references/00-context-template.md) 收集业务 / 技术 / 现有代码库与既有约定 / 系统规模 / SLA 五块。方案中已写明的直接抽取并向用户复核；缺失项标 "未说明"。

**关键 4 项缺项处理**（业务重要性 / 是否涉及资金 / 当前 QPS / SLA 目标）：
- 正式评审：任一缺失必须追问后再继续。
- 初步扫描：可临时假设并继续，但必须在报告头部列出假设与置信度。

### Step 1.5 — 外部假设核查（防"基础假设崩塌"）

> 实现阶段推翻方案，往往不是"方案内部不自洽"，而是"方案对外部现实（仓库 / 中间件部署形态 / 既有约定）的假设不成立"。本步用证据驱动的方式提前暴露这类风险。

**触发信号（任一即必须执行；都不具备可跳过，但需在报告中说明跳过原因）**：

- 方案首次引入仓库 / 团队从未使用过的中间件、命令族、能力或机制。
- 方案要替换或扩展一个既有机制（核对"现在怎么做、为什么要换"）。
- 方案关键路径依赖某外部能力提供特定语义（如"可靠投递 / 互斥 / 强一致 / 零丢失 / 全局有序"）。

**必做项（凭证据，不凭记忆）**：

1. **既有约定核查**：在仓库代码 / 注释 / 架构规范中检索方案使用的关键能力 / 命令 / 类型名，判断：是否首次引入（命中数为 0）/ 是否已被标记为 `deprecated` / `do not use` / `unreliable` / 仓库是否已有同类机制。
2. **能力假设对照**：把方案中所有"我假设 X 能做到 Y"的论断显式列出，逐条与该依赖在**本系统实际部署形态**下的真实能力比对（不是官方文档的最大能力，而是本系统配置下能拿到的能力）。
3. **替代 / 兜底盘点**：若首次引入或与既有机制冲突，盘点是否有不依赖新机制的等价方案，以及语义不满足时的补偿路径（对账 / reconcile / 全量重同步）。

核查结果在最终报告"外部假设核查"小节逐项记录：检查项 / 证据来源（含检索命令）/ 与方案假设是否一致 / 冲突点。

**冲突处置**：

- 与既有约定不一致 → Step 4 由一致性校验者按其"方案 vs 外部现实一致性"清单输出问题（[@references/01-consistency-completeness.md](references/01-consistency-completeness.md)）。
- 能力假设不成立且无兜底 → Step 4 由 HA 校验者按其"跨节点协调通道可靠性"清单输出问题（[@references/02-fault-tolerance-ha.md](references/02-fault-tolerance-ha.md)），Step 6 按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第三节"外部假设错位"规则强制升级为 Blocker。

### Step 2 — 识别方案类型（决定校验重点）

根据方案内容判定属于以下哪一种（可多选；跨类型取并集）。校验者编号详见 Step 4 表，下表使用编号引用。

| 方案类型 | 默认强制启用的校验者 | 强化重点 |
|---|---|---|
| 新系统建设 | 1 + 2 + 3 + 5 + 6 + 7 | 端到端完整性、容量评估、运维准入 |
| 老系统改造 | 1 + 6 + 7 + 8 | 兼容性、灰度、回滚、数据迁移 |
| 数据迁移 | 1 + 5 + 6 + 8 | 双写 / 影子读 / 对账、迁移期一致性、回滚 |
| API / 接口设计 | 1 + 4 + 7 + 8 | 契约、幂等、错误码、版本、鉴权 |
| 高并发活动 | 1 + 2 + 3 + 6 + 7 | 容量、限流降级、热点、预案 |
| 支付 / 交易 / 账务 | 全部启用 + 严格模式 | 资金一致性、对账、合规、审计、零丢失 |
| 消息异步化 | 1 + 2 + 5 + 6 + 8 | 可靠投递、幂等消费、积压、协议演进 |
| 存储选型 | 1 + 3 + 5 + 7 | 存储匹配度、数据建模、扩展上限 |
| 容灾改造 | 2 + 5 + 6 + 7 | RPO / RTO、切换、脑裂、回滚 |
| 权限 / 安全改造 | 1 + 4 + 6 | 越权、审计、合规、灰度 |
| 降本 / 性能优化 | 3 + 6 + 7 | 瓶颈定位、量化收益、回滚兜底 |
| 内部工具 / MVP | 1（轻量）+ 7 | 正确性 + 防过度设计 |

识别结果先告诉用户。

### Step 3 — 选择校验者组合

以 Step 2 矩阵为底，按业务重要性调档：

| 重要性 | 深度 |
|---|---|
| 核心系统 | 全部 8 个，严格模式 |
| 重要系统 | 必选组（1 + 7）+ 3~4 个按需 |
| 一般系统 | 必选组（1 + 7）+ 1~2 个按需 |
| 内部工具 / MVP | 仅 1（轻量）+ 7 |

把"本次启用了哪几个校验者、为什么"先告诉用户，允许增减。范围限定评审（如"只看性能"）下仅展开指定校验者；若发现明显跨维度 Blocker，必须在报告"额外风险提示"小节简述并建议补充评审范围。

### Step 4 — 逐个校验者执行

对选中的每一个校验者：

1. 读取对应 reference 文件（见下表），完全采用其中定义的 Role / Identity / 审查清单 / Rules——这是该校验者的 system prompt，不要改写或缩水。
2. **Output Format 统一使用 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第七节模板**（位置 / 原文证据 / 判断依据 / 修改建议 + 维度专属字段），编号前缀按下表。
3. 把 Step 1 收集的上下文注入当前校验者的视角，逐项过审查清单。
4. 给出该维度的**总结 + 评分（1–10，按 09 第六节 Rubric）**。

| # | 校验者 | Reference |
|---|---|---|
| 1 | 一致性与完整性 | [@references/01-consistency-completeness.md](references/01-consistency-completeness.md) |
| 2 | 容错与高可用 | [@references/02-fault-tolerance-ha.md](references/02-fault-tolerance-ha.md) |
| 3 | 性能与可扩展性 | [@references/03-performance-scalability.md](references/03-performance-scalability.md) |
| 4 | 安全性 | [@references/04-security.md](references/04-security.md) |
| 5 | 数据一致性与存储 | [@references/05-data-storage.md](references/05-data-storage.md) |
| 6 | 可运维性 | [@references/06-operability.md](references/06-operability.md) |
| 7 | 简洁性（奥卡姆剃刀） | [@references/07-simplicity.md](references/07-simplicity.md) |
| 8 | API 契约与兼容性 | [@references/10-api-contract-compatibility.md](references/10-api-contract-compatibility.md) |

**所有发现强制带证据**：每个问题必须包含位置、原文证据（缺失则写"方案未说明"）、判断依据、修改建议。这是合格交付的硬要求，参见 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md)。

### Step 5 — 汇总仲裁

按 [@references/08-orchestration.md](references/08-orchestration.md) 第二节的优先级排序、量化判断与不可妥协红线做最终裁决。一句话原则：**正确性 > 安全 > 数据 > 需求驱动的健壮性 > 兼容性 > 简洁性 > 非需求驱动的健壮性 / 性能**。

### Step 6 — 严重程度归一化与门禁判定

汇总所有问题前，必须执行：

1. **归一化**：按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 第二节的映射表，把各校验者原始等级（Critical/P0/High 等）统一映射到 5 档（Blocker/High/Medium/Low/Info）。
2. **Blocker 自动判定**：按 09-severity-and-gate.md 第三节规则，把符合条件的问题强制升级为 Blocker。
3. **门禁判定**：按 09-severity-and-gate.md 第四节规则，机械推导"是否可进入下一阶段"。**不允许凭主观判断给结论**。
4. **豁免登记**：High 及以下问题如不修复，必须按 09-severity-and-gate.md 第五节登记 Waiver；Blocker 不可豁免。

> 初步扫描模式跳过本步的"门禁判定"，仅做归一化和风险列举。

### Step 7 — 输出最终报告

#### 7.1 产物落盘规则（强约束）

> **本步是动作指令，不是规则描述**。Agent 必须**真正调用文件写入工具**把报告内容写到磁盘上的目标文件；只在对话里贴出报告内容**不算**完成本步。本步未完成时，Step 7 视为失败，整个评审视为未交付。

**硬动作（必做，按顺序执行，不可省略）**：

1. 计算目标落盘路径 `REPORT_PATH`（按下文"命名"规则推导）。
2. 若 `REPORT_PATH` 已存在，先按下文"重跑策略"将旧文件**重命名为备份**（也是一次真实的文件系统动作，不能只在对话里说"我会备份"）。
3. **调用可写入文件的工具**（如 `edit_file` / `write` / `replace_in_file` 等当前可用的写文件工具，取决于运行环境），把**完整的最终报告 Markdown 内容**写入 `REPORT_PATH`。
4. 在对话回复中**显式回显**：报告输出绝对路径、上一版备份绝对路径（无则写"无"）。这是给用户和 Verification 自检的可验证凭据。
5. **禁止替代行为**：不得用"我已生成报告（在下面）" / "完整报告见上方/下方" / 把整份报告贴在对话里 等方式替代真实落盘。如果当前环境**确实**没有可写文件的工具，必须**显式告知用户"无法落盘，仅以对话形式输出"**，并在 Verification 自检中标记为未达成；不得静默降级。

**命名**：报告文件名为 `<原方案文件名（不含扩展名）>-review.md`，目录为原方案文件**所在目录**。

- 例：方案路径 `docs/order-system.md` → 报告路径 `docs/order-system-review.md`。
- 例：方案路径 `/path/to/RFC-2025-001.md` → `/path/to/RFC-2025-001-review.md`。
- 方案不是来自具体文件（如直接粘贴文本）：先询问用户期望落盘目录与文件名；未指定时默认输出到当前工作目录的 `tech-review-YYYYMMDD-HHmmss.md`。

**重跑策略（备份 + 覆盖写）**：若同名 `<原文件名>-review.md` 已存在：

1. 先把旧文件重命名为 `<原文件名>-review.bak.YYYYMMDD-HHmmss.md`（时间戳为本次重跑时刻，到秒），作为备份保留在**同一目录**。
2. 以本次报告**覆盖写**原文件名 `<原文件名>-review.md`，保证"最新一次评审结果始终位于同一个稳定路径"。
3. 将备份产生的路径写到报告头部 `上一版备份路径` 字段。
4. **不追问**、**不拒写**、**不产生多份报告**。用户仅在明确表达"不需要备份"时才可跳过第 1/3 步。

> 备份文件名示例：`order-system-review.md` → `order-system-review.bak.20260521-215708.md`。

**版本号写入**：报告头部必须记录本次评审使用的 skill 版本号（取自本文件 frontmatter 的 `version` 字段，以 SKILL.md 为唯一权威来源），以便追溯"这份报告是哪个版本的 skill 跑出来的"。升级 skill 后重跑会自动使用新版本号。

**报告头部必须显式包含的元信息**：

- 原方案路径
- 报告输出路径
- 上一版备份路径（首次跑填“无”；重跑填备份文件路径）
- 评审时间（本次跑的时间，到秒）
- skill 版本号

#### 7.2 报告内容结构

按 [@templates/final-report.md](templates/final-report.md) 组织，含一个执行摘要 + 9 个正文章节：

1. **执行摘要**（Top 5 风险 + 是否可进入下一阶段 + 最大阻塞项 + 需业务方确认事项）。
2. **审查上下文摘要**（Step 1 收集的） + **假设与置信度**（初步扫描模式必填）。
3. **方案类型识别 + 本次启用的校验者列表 + 启用原因**。
4. **各校验者的发现**（按维度组织，每个问题保留原始编号如 `C-1`、`HA-3`、`SEC-2`、`API-1` 便于追溯，同时标注归一化后的统一严重程度）。
5. **冲突仲裁结论**（如果不同维度建议冲突）。
6. **整改清单**（字段化表格：编号 / 整改项 / Owner / 验收标准 / 是否阻塞 / 建议完成时间）。
7. **豁免登记表**（如有）。
8. **设计亮点**（无需调整 / 无需进一步优化的设计）。
9. **整体评分汇总表 + 门禁判定结论**（按 09-severity-and-gate.md 推导，不允许"看情况"）。

## Pitfalls

下列是高频翻车点的反向描述。**详细的"应该怎么做对"以正向 checklist 形式收敛在 Verification 自检清单中**，跑完后逐条勾选即可。

- **跑全部校验者** → 上下文爆炸 + 噪声淹没用户。按方案类型 + 特征裁剪，并解释为何裁剪。
- **冲突时不仲裁** → 用户拿到矛盾建议。Step 5 必须给出明确取舍。
- **凭印象写问题 / 不引用方案原文** → 必须引用原文作为证据；方案未说明的明确写出。
- **凭外部依赖的"理论能力"评审** → 论文上能做到，本系统部署形态下不一定能做到。出现 Step 1.5 三个信号任一项时必须走 Step 1.5。
- **跳过上下文、输出格式漂移、只列问题不给建议、严重程度跨维度漂移、门禁凭感觉、只挑刺不肯定** → 在 Verification 自检中逐条收敛。
- **Step 7 落盘三类违规**：①只在对话里贴报告不调用写工具 ②重跑不真备份就覆盖 ③报告头不写 skill 版本号 → 用户拿不到稳定路径产物 / 重跑无从 diff / 旧报告无法追溯。Step 7.1 的硬动作必须**真的执行**，并在对话里回显绝对路径。

### 反例与正例

**❌ 不合格的发现**：
> 性能可能有问题，建议优化。

**✅ 合格的发现**：
> ### [PERF-3] [瓶颈] [严重程度: High]
> **位置**：方案 §4.2 订单查询接口
> **原文证据**："订单查询通过 SELECT * FROM orders WHERE user_id=? ORDER BY create_time DESC"
> **判断依据**：用户规模 1000 万，单用户历史订单可达数千条；订单表 1 年后预估 5 亿行
> **问题描述**：未限定时间范围 + 无分页 + ORDER BY 非索引前缀，深度查询将导致 P99 飙升至秒级
> **修改建议**：1) 强制时间范围（默认近 90 天）；2) 游标分页；3) (user_id, create_time) 联合索引

## Verification

跑完后自检以下条目，全部满足才算合格交付：

- [ ] 已声明执行模式（正式评审 / 初步扫描）。
- [ ] 已记录业务重要性 / 是否涉及资金 / QPS / SLA 这 4 项核心上下文（初步扫描模式可为假设）。
- [ ] 若方案首次引入新能力 / 替换既有机制 / 关键路径依赖外部特定语义，已执行 Step 1.5 外部假设核查，并在报告中记录检查项 / 证据 / 冲突点；否则需在报告中说明跳过原因。
- [ ] 已识别方案类型，并说明启用了哪几个校验者、为何。
- [ ] 每个发现都包含：位置、原文证据（或"方案未说明"）、判断依据、统一严重程度、修改建议。
- [ ] 简洁性建议都包含：替代方案、前提条件、失效条件。
- [ ] 严重程度已按 [@references/09-severity-and-gate.md](references/09-severity-and-gate.md) 归一化为 Blocker/High/Medium/Low/Info。
- [ ] 已执行 Blocker 自动判定规则（资金/数据丢失/安全/合规/SLA 错位等）。
- [ ] 已给出字段化的整改清单（含 Owner / 验收标准 / 是否阻塞）。
